# 云开发部署指南

本文说明如何将算力贸易小程序从「本机单人演示」切换为「多人云端共享数据」。

## 一、开通云开发

1. 用微信开发者工具打开 `miniprogram-mvp` 项目
2. 点击工具栏 **云开发** → **开通**
3. 创建一个环境（建议选按量计费，测试够用）
4. 复制 **环境 ID**（形如 `cloud1-xxxx`）

## 二、配置小程序

编辑 `utils/config.js`：

```javascript
module.exports = {
  cloudEnvId: "cloud1-你的环境ID",  // 也可留空，使用 IDE 当前选中的云环境
  useCloud: true,
  useWechatPhoneAuth: false,
  defaultManualPhone: "13800138000"
}
```

**重要：** `useCloud` 必须为 `true`，否则两个账号的数据各存各的手机，无法互联。

**个人主体 / 未企业认证：** 无法使用微信授权手机号，请保持 `useWechatPhoneAuth: false`，用户在注册页手动填写 **姓名、手机号、登录密码**（无需部署 `getPhoneNumber`）。

**小程序企业认证后（可选）：** 将 `useWechatPhoneAuth` 设为 `true`，部署 `cloudfunctions/getPhoneNumber`，并在公众平台配置用户隐私保护指引。

**账号体系：** 注册时设置密码（至少6位）；登录页使用 **手机号 + 密码** 登录；「我的」页可 **退出登录**。

**同一微信多账号：** 平台以 **手机号** 区分用户，不是以微信 openid。同一部手机、同一个微信可以注册多个不同手机号的账号，但 **退出后必须先清本地缓存再登录/注册**，云端也会按手机号隔离提交记录与企业资料。若曾出现「注册 B 却看到 A 的资料」，请 **重新部署 tradeApi 云函数** 后退出并重新登录。

**勿在同一微信里用新手机号覆盖旧账号：** 旧版注册逻辑曾会把新手机号写入当前微信已绑定的运营账号，导致原运营手机号无法登录。已修复：新手机号始终 **新建独立账号**，不会继承 `staffRole`。若已发生覆盖，请在云数据库 `users` 中找回被改号的运营记录，把 `phone` 改回原号并重设密码，同时删除或改号误注册的那条记录。

## 三、创建数据库集合

在云开发控制台 → **数据库** → 新建集合：

| 集合名 | 用途 |
| --- | --- |
| `users` | 用户资料、认证状态、**收藏列表**（`favoriteResources` / `favoriteDemands`） |
| `listings` | 资源/需求公示池 |
| `submissions` | 提交记录、对接、认证 |

### 安全规则（推荐，多用户必配）

三个集合均设置为 **禁止客户端直连读写**（仅云函数可访问）：

```json
{
  "read": false,
  "write": false
}
```

完整示例见 `cloudfunctions/database-rules.json`。

所有读写必须通过云函数 `tradeApi` 完成。若设为 `read: true`，任意用户可能通过小程序直连数据库读到他人手机号等敏感信息。

### 建议索引（用户量增大后在控制台创建）

| 集合 | 索引字段 | 用途 |
| --- | --- | --- |
| `listings` | `pool` + `verification` + `publishedAt`（复合） | 公池默认列表 `listListings` |
| `listings` | `pool` + `type` + `matchScore`（复合） | 公池按类型筛选 + 完整度排序 |
| `listings` | `pool` + `region` + `publishedAt`（复合） | 公池按地域筛选 |
| `listings` | `pool` | 公池分页 `listListings` |
| `listings` | `verification` | 管理员待审公示 `adminSync` |
| `listings` | `id` | 按业务 ID 查详情 |
| `submissions` | `type` + `ownerPhone` + `updatedAt`（复合） | 对接日限额统计、sync 最近记录 |
| `submissions` | `ownerOpenid` + `updatedAt`（复合） | 用户自己的提交 sync |
| `submissions` | `type` + `phone` + `updatedAt`（复合） | 用户发起的对接 sync |
| `submissions` | `status` | 管理员待审申请 |
| `users` | `_openid` | 微信用户（系统默认） |
| `users` | `phone`（**唯一索引**） | 手机号登录 / 注册全局唯一 |

> **手机号唯一索引（强烈建议）：** 在云开发控制台 → 数据库 → `users` → 索引 → 新建索引，字段 `phone`，勾选 **唯一**。可防止并发注册产生重复手机号；若已有重复数据需先手工合并后再建索引。

> **旧版 `favorites` 集合：** 若你曾按旧文档创建过独立 `favorites` 集合，无需手工迁移。重新部署 `tradeApi` 后，用户下次 `sync` 或点星标时会自动合并到 `users` 并删除旧文档；确认迁移完成后可在控制台删除空的 `favorites` 集合。

## 四、部署云函数

在开发者工具中，分别右键以下目录 → **上传并部署：云端安装依赖**：

1. `cloudfunctions/getPhoneNumber`（仅企业主体且 `useWechatPhoneAuth: true` 时需要）
2. `cloudfunctions/tradeApi`（核心业务 API，必部署）

部署成功后，在 **云函数 → tradeApi → 日志** 中可查看调用记录。

**若运营工作台出现「云端同步异常 / Invoking task timed out after 3 seconds / -504003」：**

说明云端 `tradeApi` 仍是默认 **3 秒**超时。本项目已在 `cloudfunctions/tradeApi/config.json` 配置为 **120 秒**，但必须重新部署后才会生效：

1. 右键 `cloudfunctions/tradeApi` → **上传并部署：云端安装依赖**
2. 部署完成后在云开发控制台 → **云函数 → tradeApi → 配置** 中确认「超时时间」为 120 秒
3. 重新打开小程序，下拉刷新「我的 / 运营工作台」

所有写入通过云函数 `tradeApi` 完成，避免用户篡改他人数据。

### 批量导入演示数据（可选）

运营工作台支持一键导入 4 个测试用户及服务器整机资源/需求：

- 手机号：`18800000000` ～ `18800000003`（每个 +1）
- 密码：`123456`
- 每人 1 条整机资源 + 1 条整机需求（已初审，直接进入公池）
- 清空演示数据会删除 `18800000000`～`18800000099` 整段历史测试号（含旧版 100 用户）

操作路径：运营账号登录 → **我的（运营工作台）** → 底部 **测试数据** → **导入演示数据**。也可在云函数测试面板调用 `seedDemoData`，详见 `miniprogram-mvp/scripts/seed-demo-users.md`。

> 入口由 `utils/config.js` 的 `enableDemoSeedTools` 控制（默认 `true`）；正式上线可设为 `false` 隐藏。

## 五、数据读写说明（云端模式 · 多用户）

| 数据 | 云数据库集合 | 写入方式 | 小程序如何读取 |
| --- | --- | --- | --- |
| 用户资料 | `users` | `registerUser` / `updateProfile`（字段白名单） | `sync` 拉取本人资料 |
| 本人提交/对接 | `submissions` | `createSubmission` / `patchSubmission` | `sync` 拉取本人 + 发给自己的对接 |
| 资源/需求公池 | `listings` | `createSubmission`（发布类） | `listListings` 分页拉取（不再塞进 sync） |
| 我的收藏 | `users`（`favoriteResources` / `favoriteDemands`） | `toggleFavorite` | `sync` 拉取；点星标时即时写入云端 |
| 管理员待审 | 同上 | — | `adminSync` 拉取**全部**待审公示与申请 |
| 审核操作 | 同上 | `adminReview` | 审核后自动刷新 |

**多用户设计要点：**

1. **`sync` 只同步「我的数据」**，避免每个用户启动都拉全站几百条公示，用户多了会卡。
2. **`listListings` 按资源/需求池分页**拉公开展示数据（首页、资源页、需求页用）。
3. **普通用户 Tab 页**使用 `refreshPoolPagesFromCloud()`（本人提交 + 公池），**不**每次拉运营待审队列；运营账号进「我的/代发管理」才走 `refreshFromCloudForMine` / `refreshFromCloudFull`。
4. **`adminSync` 专供撮合经理**，可看到所有用户的认证、撮合、待审公示（不再漏单）。
5. **`updateProfile` 有字段白名单**，用户不能自行改成「已认证」。
6. **云端拉取合并以服务器为准**：`sync` / 轻量 sync 写入本地缓存时，同 id 记录用云端字段覆盖本地；仅保留「本地有、云端尚未返回」的条目（待上传）。
7. **写入云端才做冲突检测**：拉取永远以服务器为准；若本地基于旧版本提交修改，云端会返回 `stale`，客户端刷新后再操作。`updateProfile`、`patchSubmission` 均带 `_baseUpdatedAt`；对接专用接口（`agreeConnectExchange` 等）由服务端校验当前状态。

### 同步方向一览

```
┌─────────────────────────────────────────────────────────────┐
│  云端 → 本地（拉取）   合并规则：同 id 以服务器为准           │
├─────────────────────────────────────────────────────────────┤
│  公池分页      listListings / fetchPublicListings           │
│  我的/记录页   sync { lite: true } → refreshUserLiteFromCloud│
│  运营工作台    staffWorkbenchSync { lite: true }            │
│  写操作后后台  scheduleBackgroundCloudSync → 轻量 sync      │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  本地 → 云端（推送）   成功后用服务端返回值更新本地缓存       │
├─────────────────────────────────────────────────────────────┤
│  注册/登录     registerUser / loginUser                     │
│  改资料        updateProfile（_baseUpdatedAt 冲突检测）     │
│  发布/对接     createSubmission                             │
│  改状态/关闭   patchSubmission（_baseUpdatedAt 冲突检测）   │
│  交换名片等    agreeConnectExchange / confirmConnect…       │
│  收藏星标      toggleFavorite → users.favoriteResources/Demands │
│  运营审核      adminReview                                  │
└─────────────────────────────────────────────────────────────┘
```

**冲突何时发生：** 只在「本地 → 云端」写入时。例如 A 手机改了对接状态但未同步，B 手机已改同一记录；A 再提交时云端 `updatedAt` 更新，返回 `stale`，A 本地会被刷新为云端最新版，需重新操作。

**仍保留全量 `sync` 的场景：** 首次迁移 `migrateLocalToCloudIfNeeded`、显式调用 `refreshFromCloud()`（内部带迁移）。日常页面不再走全量 sync。

## 六、云函数能力说明

`tradeApi` 支持以下 action：

| action | 作用 |
| --- | --- |
| `sync` | 拉取当前用户资料 + 相关提交（不含公池） |
| `listListings` | 分页拉取资源/需求公池 |
| `adminSync` | 管理员拉取全部待审公示与申请 |
| `registerUser` | 注册用户（姓名、手机号、密码） |
| `loginUser` | 手机号密码登录，绑定当前微信 openid |
| `updateProfile` | 更新企业资料（白名单字段） |
| `createSubmission` | 创建提交（含发布公示） |
| `patchSubmission` | 更新状态、对接确认、关闭需求（带 `_baseUpdatedAt` 冲突检测） |
| `toggleFavorite` | 收藏/取消收藏（按 listingId + favorited 幂等写入） |
| `adminReview` | 撮合经理审核通过/驳回 |
| `adminTakeDownListing` | **平台管理员**强制下架已发布资源/需求 |
| `adminDisableAccount` | **平台管理员**禁用普通用户账号（并下架其商机） |
| `adminEnableAccount` | **平台管理员**解除账号禁用 |
| `adminSearchPublishedListings` | **平台管理员**搜索公开展示商机 |
| `adminLookupUser` | **平台管理员**按手机号查询用户 |
| `migrateLocal` | 旧版本地数据一次性迁移 |

## 七、多人使用验证

1. 手机 A：注册（设置密码）→ 发布一条资源
2. 手机 B：注册或登录 → 在资源池应能看到 A 发布的资源
3. 手机 B：申请对接 → 手机 A（资源方）在「提交记录」中确认
4. 撮合经理：登录审核工作台 → 审核通过/驳回（结果所有用户可见）

## 八、本地模式 vs 云端模式

| 对比 | 本地模式 `useCloud: false` | 云端模式 `useCloud: true` |
| --- | --- | --- |
| 数据存储 | 手机本地 | 云数据库 |
| 多人共享 | 不支持 | 支持 |
| 对接确认 | 需同一台手机切换账号 | 不同用户各自操作 |
| 切换账号 | 退出登录后本机缓存清空 | 退出登录 → 用另一手机号登录；本地仅缓存当前账号 |

## 九、常见问题

### 弹窗「云端数据未联通」

按顺序排查（多数情况是前 3 步未完成）：

1. **开通云开发**：开发者工具顶部 → 云开发 → 开通
2. **填写环境 ID**：云开发控制台复制 `cloud1-xxxx`，写入 `utils/config.js` 的 `cloudEnvId`  
   - 模拟器可留空并用工具栏选环境；**真机预览、两台手机互测必须填写**
3. **建集合**：数据库 → 新建 `users`、`listings`、`submissions`（名称完全一致）
4. **部署云函数**：右键 `cloudfunctions/tradeApi` → **上传并部署：云端安装依赖**
5. 编译重新运行小程序

### 提示「请部署 tradeApi 云函数」

说明云函数未部署或环境 ID 填错，请检查 `cloudEnvId` 并重传云函数。

### 模拟器看不到他人数据

云开发需联网；请确认两台设备都已 `useCloud: true` 且部署成功，进入页面时会自动 `sync`。

### 认证图片无法预览 / 运营专员看不到图

用户上传的名片、营业执照会存到 **云存储**（`cloud://` 开头的地址）。运营专员必须用 **已注册并开通运营权限的微信账号** 进入工作台，这样云 API 会带上当前微信的 **openid**，才能正常换取图片链接。

**推荐做法（运营专员账号）：**

1. 让同事先用手机号 **注册/登录** 普通用户（确保 `users` 集合里有记录且绑定了 openid）
2. 在云开发控制台 → **数据库** → `users` → 找到该用户记录，新增字段：
   - `staffRole`: `"manager"`（运营专员）或 `"admin"`（平台管理员）
   - 也可在 `cloudfunctions/tradeApi/index.js` 的 `STAFF_PHONE_WHITELIST` 填入手机号后重新部署云函数，用户下次登录会自动开通
3. 用户打开小程序 → **我的** → **其他** → **平台运营**（不再使用独立的 manager/2580 演示账号）
4. 在运营中心进入对应审核功能，认证图片应可正常显示

**云存储权限（仍需配置）：**

1. 微信开发者工具 → **云开发** → **存储** → **权限设置**
2. 选择 **所有用户可读，仅创建者可写**（推荐），保存

若仍看不到，在云开发控制台 → **存储** → **文件管理** 中确认 `cert/` 目录下是否有文件；若没有，说明用户提交时上传失败，需让用户重新上传认证材料。

## 十、正式上线检查清单

上线前请逐项确认，避免演示配置或安全缺口流入生产环境：

| 检查项 | 推荐值 | 说明 |
| --- | --- | --- |
| `config.js` → `autoApproveListing` | `false` | 商机须经运营审核后进公池 |
| `config.js` → `enableDemoSeedTools` | `false` | 隐藏运营工作台「导入/清空演示数据」入口 |
| 云函数环境变量 `DEMO_SEED_TOOLS_ENABLED` | 不设置或 `false` | 即使运营账号也无法通过 API 注入演示数据 |
| `config.local.js` → `legalOperatorName` / `legalContactEmail` | 真实主体信息 | 协议与隐私政策须可联系 |
| `users.phone` 唯一索引 | 已创建 | 防并发重复注册 |
| `submissions` 复合索引 | `type + ownerPhone + updatedAt` 等 | 保障 sync 按最近记录排序 |
| 数据库安全规则 | 三集合均 `read/write: false` | 禁止客户端直连敏感数据 |
| `tradeApi` 超时 | 120 秒 | 多人 sync / 公池拉取 |

**已知产品边界（非缺陷，但须运营知晓）：**

- 无短信验证码：手机号由用户自行填写，平台不做实名校验（企业认证后可选微信授权手机号）
- 无在线支付与交易担保：撮合止于名片交换，资金往来线下完成
- 单账号对接记录超 200 条时，sync 仅保留最近部分，「我的」页会提示
- 公开展示池超过约 1000 条时，`listListings` 可能截断，须尽快做服务端筛选分页

## 十一、下一步优化
